## Module Infrastructure (5/5)

- [x] **INFRA-01**: Compiler discovers all `.snow` files in a project directory recursively
- [x] **INFRA-02**: File paths map to module names via convention (`math/vector.snow` -> `Math.Vector`, snake_case -> PascalCase)
- [x] **INFRA-03**: Each `.snow` file is parsed independently into a per-module AST
- [x] **INFRA-04**: `snowc build <dir>` compiles all discovered `.snow` files as a unified project (not just `main.snow`)
- [x] **INFRA-05**: `main.snow` in project root is the entry point containing `fn main()`, not treated as a module

## Import Resolution (7/7)

- [x] **IMPORT-01**: `import Math.Vector` brings `Vector` into scope as a qualified namespace (`Vector.add(a, b)`)
- [x] **IMPORT-02**: `from Math.Vector import { add, scale }` injects specific names into local scope unqualified
- [x] **IMPORT-03**: Compiler builds module dependency graph from import declarations
- [x] **IMPORT-04**: Modules are compiled in topological sort order (dependencies before dependents)
- [x] **IMPORT-05**: Circular imports produce a compile error with the cycle path (`A -> B -> C -> A`)
- [x] **IMPORT-06**: Importing a non-existent module produces a clear error with file path suggestion
- [x] **IMPORT-07**: Importing a non-existent name from a module produces a clear error

## Cross-Module Semantics (7/7)

- [x] **XMOD-01**: Functions defined in one module can be called from another module via qualified access (`Vector.add(a, b)`)
- [x] **XMOD-02**: Functions imported selectively can be called unqualified (`add(a, b)` after `from Math.Vector import { add }`)
- [x] **XMOD-03**: Struct types defined in one module can be constructed and used in another module
- [x] **XMOD-04**: Sum types defined in one module can be pattern-matched in another module
- [x] **XMOD-05**: Trait impls are globally visible across all modules (not selectively imported)
- [x] **XMOD-06**: Generic types and functions work correctly across module boundaries (monomorphization)
- [x] **XMOD-07**: Two modules can each define private functions with the same name without collision

## Visibility (5/5)

- [x] **VIS-01**: All items (fn, struct, sum type, interface) are private by default
- [x] **VIS-02**: `pub` modifier makes an item visible to importing modules
- [x] **VIS-03**: Accessing a non-pub item from another module produces a compile error with `pub` suggestion
- [x] **VIS-04**: All fields of a `pub struct` are accessible to importers (no per-field visibility in v1.8)
- [x] **VIS-05**: All variants of a `pub type` (sum type) are accessible to importers

## Diagnostics (3/3)

- [x] **DIAG-01**: Error messages for cross-module issues include the source module name and file path
- [x] **DIAG-02**: Type errors involving imported types show the module origin (e.g., "expected Math.Vector.Point")
- [x] **DIAG-03**: Single-file projects continue to compile identically (full backward compatibility)

## Future Requirements (Deferred)

- Implicit prelude module

## Out of Scope

- Glob imports (`from Module import *`) -- explicitly rejected in parser, obscures name origins
- Circular module dependencies -- destroys modularity, complicates compilation ordering
- First-class modules (modules as values) -- massive type system complexity, use traits instead
- Functors (parameterized modules) -- use generic types and traits instead
- Module signatures/interface files (`.snowi`) -- `pub` items define the interface
- Dynamic/runtime imports -- incompatible with static compilation and HM inference
- Module-level mutable state -- violates immutability-first design, use actors
- Relative imports (`from ../utils import helper`) -- fragile, use absolute module paths
- Orphan rules -- defer until package ecosystem exists
